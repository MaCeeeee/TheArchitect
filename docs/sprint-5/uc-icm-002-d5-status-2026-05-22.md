## Was blockiert

### Hostinger-Deploy-Step (run 26294269683 job `deploy`)

```
curl -sf -X POST \
  -H "Authorization: ***" \
  -H "Content-Type: application/json" \
  -d "$PAYLOAD" \
  "https://developers.hostinger.com/api/vps/v1/virtual-machines/$HOSTINGER_VM_ID/docker"
##[error]Process completed with exit code 22.
```

`curl -sf` exit 22 = HTTP 4xx/5xx vom Hostinger-API. `-s` blendet
den Response-Body aus → kein konkreter Error sichtbar.

**Mögliche Ursachen:**
1. `HOSTINGER_API_KEY` ist abgelaufen / rotiert
2. `HOSTINGER_VM_ID` falsch
3. API-Endpoint-Schema von Hostinger seit März 2026 geändert
4. Image-Pull-Probleme (private GHCR — VPS-Zugriff?)

### Auth-Gate verhindert HTTP-Smoke-Test der neuen Endpoints

Production läuft `authenticate` Middleware **vor** der Route-Resolution.
Daher gibt `/api/projects/X/compliance/mappings/auto` **401** zurück —
ob die Route existiert oder nicht. Pre-Deploy- und Post-Deploy-Smoke
sehen identisch aus.

Smoke-Probe-Output (Baseline VOR Deploy):
```
✅ HTTP 200  app health
✅ HTTP 401  W1 regulations crawler-health (existed pre-W2)
✅ HTTP 401  W2 compliance/mappings/auto
✅ HTTP 401  W2 compliance/mappings/preview
✅ HTTP 401  W2 compliance/mappings/by-element
✅ HTTP 401  W2 compliance/mappings/by-regulation
✅ HTTP 401  W2 compliance/mappings/confirm
```

⚠️ **Diese 7/7 PASS beweisen NICHTS** — bogus routes geben auch 401 zurück.

## Was als nächstes getan werden muss

### Option 1 — Hostinger-API-Pfad reparieren (User-Action)

1. Prüfen ob `HOSTINGER_API_KEY` noch gültig (GH-Secrets):
   ```bash
   curl -sS -X GET \
     -H "Authorization: Bearer $KEY" \
     "https://developers.hostinger.com/api/vps/v1/virtual-machines"
   ```
2. Falls 401: neuen API-Key in [Hostinger Console](https://hpanel.hostinger.com) generieren
3. GH-Secret aktualisieren:
   ```bash
   gh secret set HOSTINGER_API_KEY -R MaCeeeee/TheArchitect
   ```
4. Workflow neu triggern:
   ```bash
   gh workflow run "Build & Deploy" --ref master
   ```
