## Design-Entscheidung (vor Task 1 bestätigen)

**`criticality` + `traceTarget` leben auf `ComplianceRequirement` (Instanz), nicht in einem neuen Template-Store.**
- Begründung: `Regulation` UND `ComplianceRequirement` sind im bestehenden Schema bereits `projectId`-scoped (UC-ICM crawlt pro Projekt). Der Art.-30-Satz wird also **pro Projekt geseedet** — konsistent mit dem vorhandenen Muster, niedrige Change-Amplification.
- **YAGNI:** Eine Template-Ebene (kanonischer Satz, projektunabhängig, für DORA/NIS2-Skalierung) wird **bewusst aufgeschoben**. Erst extrahieren, wenn die 2. Regulierung real ansteht — sonst vorzeitige Abstraktion (Ousterhout: Komplexität durch Abhängigkeiten).
- **Komplexitäts-Watch-Point:** `traceTarget` als Mini-DSL = die eine neue kognitive Last. Mitigation: die 3 strukturierten Beispiele aus THE-352 als Doc-Kommentar am Typ.

```ts
// compliance.types.ts — neben ComplianceRequirementDTO
export type Art30Criticality = 'HART' | 'BEDINGT' | 'WEICH';

export interface TraceStep {
  rel: string;                       // ConnectionType, z.B. 'assignment' | 'realization' | 'access'
  to: string;                        // ElementType oder '*'
  where?: Record<string, unknown>;   // Knoten-Constraints, z.B. { kind: 'Purpose' }
}

/**
 * Maschinenlesbarer Trace-Pfad für REQ-WFCOMP-001.4 (THE-355).
 * Beispiele (siehe THE-352):
 *   a: { from:'process', steps:[{rel:'assignment', to:'business_role', where:{role:'Controller'}}] }
 *   e (BEDINGT): { from:'process', guard:{flag:'thirdCountry',equals:true},
 *        steps:[{rel:'flow',to:'business_role',where:{role:'Recipient'}},{rel:'association',to:'*',where:{kind:'Safeguard'}}] }
 *   g (cross-layer): { from:'data_object', where:{personal:true},
 *        steps:[{rel:'access',to:'application_component'},{rel:'serving',to:'node'},{rel:'association',to:'requirement',where:{kind:'TOM',art32:true}}] }
 */
export interface TraceTarget {
  from: string;
  where?: Record<string, unknown>;
  guard?: { flag: string; equals: unknown };  // bedingt (lit. e): greift nur wenn flag === equals
  steps: TraceStep[];
}
```
Und in `ComplianceRequirementDTO` ergänzen:
```ts
  criticality?: Art30Criticality;   // WFCOMP: Art.-30-Klasse (HART/BEDINGT/WEICH)
  traceTarget?: TraceTarget;        // WFCOMP: erwarteter Graph-Pfad
```

## Chunk 2 (detailliert, nach Read 2026-06-27): In-Memory Compliance-Pipeline

### Architektur-Refinement — die eine Fork (aus dem Read)

Zwei Befunde drehen die Architektur:
1. **n8n wird CLIENT-seitig geparst** (`client/src/utils/n8nParser.ts` → POST `{elements,connections}` an `/:projectId/import/n8n`). Der Server sieht `parameters`/`pinData`/`credentials` NIE. Ein server-seitiger Lift auf dem importierten Graphen hätte keine Domains/Feld-Namen mehr.
2. **Neo4j wird in 23 Test-Files gemockt** (`jest.mock('../config/neo4j')`), nie real gefahren.

⇒ **M1 = reine In-Memory-Pipeline** `assessWorkflow(rawN8nJson): AssessmentResult`, komponiert aus sanitize → scope → lift → trace. **DB-frei testbar** gegen die Fixtures. Server-seitig (für API/MCP-Pfad ohnehin nötig). **Neo4j-Persistenz + Cypher-Ausführung = dünner Adapter NACH M1** (Integration, nicht Mechanismus) → eigenes Follow-up-Issue.

**Begründung (Ousterhout):** Die Trace-Logik als reine, getestete Funktion über ein Graph-Modell ist die Mitigation des Trace-DSL-Watch-Points — obvious, testbar, ohne Neo4j-Abhängigkeit im Mechanismus. Genau die „Abhängigkeiten reduzieren"-Regel.

Neues Modul `packages/server/src/services/wfcomp/`: `types.ts`, `sanitize.ts`, `scope.ts`, `lift.ts`, `trace.ts`, `assess.ts`.

### Task 4 (.0 / THE-358): `sanitizeN8nWorkflow` — schaltet G1

- `sanitizeN8nWorkflow(raw): SanitizedWorkflow` — liest rohes n8n-JSON, gibt struktur-only zurück:
  `{ name, nodes: [{ name, type, paramKeys: string[], targetDomains: string[] }], edges: [{from,to,kind}] }`.
  - **Verwirft:** `pinData` (ganz), `credentials`, Parameter-**Werte**. **Behält:** Parameter-**Keys** (z.B. `email`,`iban`), Node-Typ/Name/Topologie.
  - `targetDomains`: aus URL-Parametern nur den **Hostname** (kein Pfad/Query → „Domains, nicht Payloads", AC-4).
- TDD: Test gegen `pindata-leak.json` → `JSON.stringify(sanitized)` enthält KEINEN der PII-Werte (`hans.mueller@…`, `Erika Musterfrau`, `DE89…`), aber `paramKeys` enthält `email`/`fullName`. **= G1.**

### Task 5 (.2 / THE-353): `detectGdprScope` + `liftCompliance` — schaltet G5 + den Lift

- `detectGdprScope(sanitized): boolean` — personenbezogen, wenn PII-Feld-Keys (email/name/iban/phone/…) ODER Collection-Trigger→Store. `no-personal-data` → `false`. **= G5.**
- `liftCompliance(sanitized): LiftedGraph` — GDPR-Semantik-Graph (in-memory), `provenance:'import'`:
  - Trigger (webhook/form) → Collection-Point; Set-Keys mit PII → `data_object{personal:true}`.
  - HTTP/SaaS mit externem Hostname → `business_role{Recipient}`; non-EU-Domain → `thirdCountry:true`-Kandidat.
  - DB-Node → `application_component` (Storage) → `node`.
- TDD: `clean-compliant` liftet einen Recipient (cleverreach.de); `missing-recipient` keinen.

### Task 6 (.4 / THE-355): `runTraceCheck` (rein) — schaltet G2/G3

- `runTraceCheck(lifted, requirements): GapReport` — pro Requirement mit `traceTarget`: ab `from` (matching `where`) die `steps` traversieren, `guard` respektieren. Status je Feld: `present`(grün) | `missing`(rot) | `needs_attestation`(gelb/pending).
  - **Deterministisch entscheidbar in M1:** d (Recipient), e (Drittland-Guard). Lift-unproduzierbare Felder (a Controller, b Purpose, c Kategorie, f Retention, g TOM-Adäquanz) → `needs_attestation` wenn absent (NIE auto-rot, NIE auto-grün → M2/Attestierung).
  - Multi-DataObject: `all(d IN dataObjects WHERE …)`-Semantik (in-memory), kein false-ok durch ein abgedecktes Nachbar-Objekt.
- `assess.ts`: `assessWorkflow(raw) = sanitize → scope → (scope? lift→trace : not-applicable)`.
- TDD: `missing-recipient` → d `missing` (rot); `clean-compliant` → d `present`, a/b/c `needs_attestation` (NIE grün) **= G2 + G3**.

### Task 7: Guard- + Multi-Object-Fixtures + G4

- Neue Fixtures: `thirdcountry-no-safeguard` (e rot), `thirdcountry-with-safeguard` (e grün), `multi-dataobject-partial-tom` (g-Storage-Pfad: ein DataObject ohne → kein false-ok). Ground-Truth ergänzen. **= G4.**

### Task 8: Eval-Harness scharf schalten (G1–G5 von `todo` → real)

- Die 5 `it.todo` in `wfcomp-eval.test.ts` durch echte Assertions ersetzen, die `assessWorkflow(fixture)` gegen `GROUND_TRUTH` prüfen. **M1 fertig ⟺ G1–G5 grün.**

### NACH M1 (Follow-up-Issue, nicht M1): Neo4j-Adapter
- `liftCompliance`-Output persistieren + `runTraceCheck` gegen Cypher auf `:ArchitectureElement{type}`/`:CONNECTS_TO{type}` (Mock-Pattern wie die 23 Files). Mechanismus bleibt die reine Pipeline; Cypher ist nur die Produktions-Ausführung.
