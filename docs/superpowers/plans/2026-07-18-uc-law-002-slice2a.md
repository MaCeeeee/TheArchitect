> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Aus den deterministischen Slice-1-Kandidaten ein governtes, getractes **LLM-Applicability-Urteil** je Gesetz erzeugen (THE-462), es mit dem Stage-A-Report zu EINEM Report mit Provenance `rules|corpus|both` mergen und mit **Lifecycle `auto→confirmed|rejected`** persistieren (THE-463) — alles hinter dem bestehenden `LAW_DISCOVERY_ENABLED`-Flag. Kein UI (Slice-2b), keine Evals-Golden-Sets (Slice-2b/THE-465).

**Architecture:** `lawJudge.service` spiegelt `complianceJudge.service` (Tool-Use-JSON, Zod, 2× Retry, Anti-Halluzination, best-effort AiTrace) — Judge-Modell Haiku-4.5-Default (env→Sonnet, Owner 2026-07-18). Judge läuft nur über Retrieval-Schwelle, gecacht je `(profileHash, family, corpusVersionHash)`. `LawDiscoveryFinding` (neues Mongo-Model, family-Level) persistiert Befunde mit Lifecycle (Muster `ComplianceMapping`). `mergeApplicability` fügt Stage A (deterministisch, autoritativ) + Stage B (Korpus/Judge) zu einem `ApplicabilityReport` mit additivem `provenance`/`corpus`-Feld je Assessment; Join-Key = `family` ↔ `ruleId`. `POST /norms/discover` wird erweitert (Judge+Merge+Persist), `POST /norms/discover/confirm` + `/reject` neu.

**Tech Stack:** TypeScript strict, Node/Express, Jest, `@anthropic-ai/sdk`, Zod, Mongoose. LLM via injizierbarem Anthropic-Client (Tests ohne Netz).

> **`corpusVersionHash` — Definition (Review-Fix 1):** Es gibt KEINEN globalen Korpus-Versions-Skalar; `getCurrentVersionHashes(keys)` liefert einen Hash PRO `regulationKey`. Eine Familie aggregiert mehrere Sources/Paragraphen. Daher ist der Dedup-Hash ein **abgeleiteter Evidence-Set-Hash** über die tatsächlich gesehene Evidenz des Kandidaten:
> `computeVersionHash(candidate.topHits.map(h => \`${h.regulationKey}:${h.versionHash}\`).sort().join('|'))` (nutzt den bestehenden `computeVersionHash` aus `utils/regulationVersion`). `CorpusHit` trägt `versionHash` (Slice-1). Bewusste Konsequenz: ändert sich der Paragraphen-Inhalt/-Version, ändert sich der Hash → neuer Befund. Der Hash hängt an den `topHits` (Top-K je Kandidat); da Slice-1-Retrieval bei unverändertem Korpus stabil sortiert, ist er reproduzierbar.
