## Design-Entscheidung (vor Bau bestätigen): Status/Mode → Drei Listen

| Liste | Bedeutung | aus GapReport |
|---|---|---|
| 🟢 **Abgedeckt** | erfüllt | `status: 'present'` · Badge: `machine` (litera d/e) vs. `attested` (a/b/c via Unterschrift) |
| 🟡 **Zu bestätigen** | LLM-Vorschlag wartet auf Bestätigung | `needs_attestation` + `mode:'confirm'` (zeigt `suggestion`) |
| 🔴 **Offen** | braucht Aktion | `status:'missing'` (Lücke + Empfehlung) **oder** `needs_attestation` + `mode:'ask'` (Eingabe nötig) |

- **„Was es noch braucht"** = Gelb (unbestätigt) + Rot. (Success-Doc.)
- **G11 „Vollständig"-Regel:** vollständig ⟺ JEDES HART- und BEDINGT-Feld hat `status:'present'`. WEICH darf `needs_attestation` bleiben (begründungspflichtig, nicht blockierend).
- **G12 Honest-Naht:** strukturell-fertig-aber-unbestätigt → Header zeigt „Struktur geprüft — N Felder brauchen Ihre Bestätigung/Angabe", **nie** „konform" ohne Attestierung, **nie** ein aggregiertes %.
- **G10:** drei getrennte Listen, nach Kritikalität (HART/BEDINGT/WEICH) gruppiert, Provenance-Badge je Item. KEIN Score.

Green-Sub-Badge wird aus `litera ∈ {d,e}` abgeleitet (machine) vs. sonst (attested) — keine neue Datenstruktur nötig.
