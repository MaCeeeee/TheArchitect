# REQ-LAW-002.7 — HyDE-Query-Übersetzung ins Prod-Discovery-Retrieval — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Die HyDE-Query-Übersetzung (Architektur-Profil → LLM-Hypothese in Juristen-Sprache → embedden → Vektor-Suche) aus dem Offline-Eval-Harness in den **Prod-Discovery-Pfad** heben, flag-gegated und dark by default — der gemessene +56pp-Recall-Hebel gegen den Vokabular-Graben.

**Architecture:** Ein neues geteiltes Modul `services/hyde.service.ts` (`hydeRewrite` + `HYDE_INSTRUCTION`) wird **die eine Prompt-Quelle** für Offline-Precompute UND Prod. In `lawDiscovery.service.ts` `discoverCandidates` sitzt ein flag-gegateter Einschub zwischen Profil-Bau und `governedCorpusSearch`: wenn `LAW_DISCOVERY_HYDE=true` UND ein Provider da ist, ersetzt die HyDE-Hypothese den Query-Text; jeder Fehler/keyless fällt **graceful auf den Baseline-Profiltext** zurück (Discovery bricht nie). Der Embed-Pfad (Sidecar in `governedCorpusSearch`) bleibt unberührt — HyDE liefert nur einen Ersatz-String. **Replace-Semantik** (matcht die +56pp-Messung).

**Tech Stack:** TypeScript (packages/server), Anthropic SDK (Haiku via `defaultJudgeModel()`), Jest.

**RVTM:** docs/superpowers/rvtm/2026-07-19-the-514-hyde-prod-rvtm.md · **Linear:** THE-514 (Parent THE-459) · Score 80,0.

## Design-Entscheidungen

**DD-1 — Fallback-Policy im Caller, nicht im Helper.** `hydeRewrite` bleibt rein: gibt den getrimmten Hypothesen-Text zurück, **wirft** bei leerer/fehlgeschlagener Antwort (wie die Eval-Version). Die *Graceful-Fallback-auf-Baseline*-Policy lebt im Caller (`discoverCandidates`): `try { queryText = await hydeRewrite(...) } catch { queryText = profile.text }`. So ist die Discovery-Robustheit an einer Stelle sichtbar und der Helper testbar isoliert.

**DD-2 — Provider-Guard vor dem Call.** HyDE wird nur versucht, wenn `hydeEnabled()` UND ein Provider verfügbar ist (injizierter Client ODER `ANTHROPIC_API_KEY`) — dieselbe `hasProvider`-Logik wie beim Judge (`discoverAndJudge` L165). Keyless → HyDE übersprungen, Baseline-Text, kein Fehler.

**DD-3 — Eine Prompt-Quelle (DRY).** `HYDE_INSTRUCTION` + `hydeRewrite` leben in `hyde.service.ts`. `build-discovery-eval-vectors.ts` importiert sie statt der eigenen script-privaten Kopie — sonst driften Offline-Golden und Prod-Prompt auseinander (die +56pp-Messung wäre dann nicht mehr repräsentativ). Der Refactor muss **prompt- und verhaltensgleich** bleiben (die Eval-Precompute-Semantik unverändert).

**DD-4 — Modell + Budget.** HyDE nutzt `defaultJudgeModel()` (Haiku-Default, ein Modell-Knopf mit dem Judge) und einen **eigenen** `HYDE_MAX_TOKENS = 400` (NICHT das 2048er-Judge-Budget). Ein User-Turn, kein System-Prompt, keine Tools (wie die Eval-Version).

**DD-5 — Replace, nicht Hybrid.** Der Slice ersetzt den Query-Text durch die Hypothese (matcht die gemessene +56pp). Hybrid (baseline ∪ HyDE, zwei Suchen mergen) ist ein **Tuning-Follow-up**, nicht hier.

**OUT (bewusst):** HyDE default-on (bleibt dunkel bis Prod-Recall gemessen) · `provisionKind`/Korpus-Seite (THE-421) · HyDE-Caching · Hybrid-Retrieval · Änderung der Golden-Vektoren (der Eval-Refactor ist prompt-gleich).
