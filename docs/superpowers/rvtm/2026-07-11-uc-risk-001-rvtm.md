## 1. Prinzip: eine Engine, zwei Linsen

RISK (proaktiv, „was könnte schiefgehen") und PROBMGMT (reaktiv, „was ist schiefgegangen") teilen ~80 % Infrastruktur: **ein** WORM-Register (diskriminiert per `kind`), **eine** deterministische Score-Matrix-Mechanik, **ein** Human-Gate, **ein** Closed-Loop, **ein** n8n-Ingest-Kontrakt. Die RISK-REQs bauen **nur das risiko-spezifische Delta** und erben den Rest aus THE-445 (PROBMGMT-001.1).

Delta gegenüber der reaktiven Linse:
- Score = **Likelihood × Impact** (statt `Severity×Urgency×Criticality − Mitigation`)
- Routing = **Risk Appetite** (statt Kritisch/Normal/Noise)
- Gate-Kern = **Risiko-Akzeptanz** braucht Sign-off (Asilomar #16)
- Closed Loop = **Residual-Re-Score** (statt Verify-Closure des Fixes)
- Proaktiver Trigger = **Cron auf schleichende Risiken** (die reaktive Linse hat keinen)

## 4. Vier Korrekturen an der Ursprungs-Skizze (im Register verankert)

1. **Determinismus:** Score = versionierte Matrix (REQ-.1 AC-2), LLM nur Vorschlag (REQ-.4 AC-2).
2. **Asilomar #16:** Keine Auto-Akzeptanz — Accept-Gate mit Sign-off (REQ-.1 AC-4); Notify = Template mit Bestätigung (REQ-.4 AC-3).
3. **Closed Loop:** Residual-Re-Score + SLA-Breach-Eskalation (REQ-.3).
4. **Stabiler Fingerprint:** `hash(source + ghsa_id/cve_id + affected_component)`, nicht Freitext-Titel (REQ-.2 AC-2).
