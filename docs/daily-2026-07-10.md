# 2026-07-10 — Kanonisierungs-Trilogie: „Ein neues Gesetz ist eine Datenzeile, kein Code-Edit"

## TL;DR

Drei PRs, ein rotes Prinzip, konsequent durch drei Schichten getrieben: **Quellen-/Enum-Wildwuchs kollabiert auf die Norm-Ontologie — ein neues Gesetz/eine neue Quelle wird zur Datenzeile statt zum Code-Edit.** Erst **THE-413 (PR #42)**: 7 handgepflegte Enum-Kopien auf `normSources` (Ontologie v1.1.0) zusammengeführt — und dabei fiel die stille **THE-396-Gate-Lücke** auf (ai-act/data-act wurden von Route + Scheduler heimlich verworfen). Dann **THE-414 (PR #43)**: zwei hartcodierte Source-Registries + `CrawlBodySchema`-`z.enum` auf **eine** datengetriebene Registry, mit **Provenance pro ingestiertem Fakt** am Write-Site. Schließlich **THE-417 (PR #44)**: Ontologie **v1.2.0** (neue `languages`-Facette + `framework`/`custom` normKinds), Sprach-Enums kollabiert, **`ontologyVersion` pro Node gestempelt**, und der lange offene **Norm-Write-Pfad erzwungen** (schließt die THE-390-P0-Pflicht: Validators + `runValidators` auf `upsertNormDoc`). Alles rein additiv/ontologie-getrieben, `tsc` sauber, turbo build 4/4 grün. **Das Fundament, das heute THE-418 „DORA mit null Code" erst ermöglicht hat.**

## Teil 1 — THE-413: Source-Registry als Daten, Enum-Triplikation entfernt (PR #42)

Der Kern von **REQ-CANON-001.1** (ADR-0004 E6): Quellen-Identität lebt in der Ontologie, nicht in verstreuten Enum-Kopien.

- **7 handgepflegte Enum-Kopien** — `RegulationSource`, beide `PolicySource`-Kopien, `RegulationSourceKey`, `VALID_SOURCES` — **auf `normSources` (Ontologie v1.1.0) zusammengeführt.** Eine neue Quelle = eine Datenzeile, kein Code-Edit.
- Mongoose-`source`/`jurisdiction`-Felder validieren über `isNormSource`/`isJurisdiction` — **null-tolerant für Built-in-Enum-Parität** (Review-Fund: Custom-Validators lehnen `null` ab, native Enums nicht → bestehende `Policy`-Docs mit `source: null` bleiben gültig).
- **Behebt die THE-396-Gate-Lücke:** `ai-act-*` / `data-act-*` passieren jetzt die Server-Route + Scheduler-Gates — vorher wurden sie **still verworfen** (RED gegen die alten Gates beobachtet → GREEN).
- wfcomp-Key-Utility-Replik gelöscht (Byte-Stabilität via `key-stability.test.ts` gepinnt) → schließt das **THE-368-TODO**.
- AC-4-Kontinuität: THE-390-Facade-Suite unberührt grün + neuer section-level `regulationKey`-Assert.
- **Verifikation:** Datengetriebene Schema-Tests iterieren die Ontologie — Server 22 + Crawler 15, keine hartcodierten Listen. **Grep-Beweis: 0 Treffer** für `RegulationSourceKey` / Model-Source-Enums / `VALID_SOURCES` / `'nis2'` in shared types. Crawler-Suite 103/103, `tsc` sauber, turbo 4/4. (Die 10 `remediation.test.ts`-Fails = vorbestehende Live-Server-Integration, unabhängig.)

## Teil 2 — THE-414: Config-getriebene Source-Registry + Provenance (PR #43)

**REQ-CANON-001.2** (am 10.07. neu geschnitten): die Verdrahtungs-Ebene wird ebenfalls Daten.

- **Zwei hartcodierte Registries** (`crawl.ts buildSourceRegistry` + `cli/crawl-live.ts SOURCES`) **+ das `CrawlBodySchema`-`z.enum`** auf **eine** datengetriebene Registry (`sources/source-registry.ts`) kollabiert; Source-Validierung über `isNormSource` (THE-413-Ontologie). Eine Quelle, die einen bestehenden Adapter wiederverwendet, ist **ein Dateneintrag**.
- **Provenance** (`adapter` / `format` / `fetchedAt` / `sourceUri`) **pro ingestiertem Fakt** am Write-Site aus dem Registry-Eintrag gestempelt; optionales Sub-Doc auf beiden Regulation-Models (**UC-PROV-Hook** — schließt an die Trust-Spine an).
- Review-Fixes: CLI bleibt **always-direct** (Selector-Verifikation + THE-403-Firecrawl-Credit-Budget respektiert); ein einziger Registry-Lookup in der Route.
- Bestehende 7 Quellen: Factory-Calls **byte-identisch** (gegen gelöschten Code verifiziert), Response-Shape unverändert; einziges Delta: togaf/archimate liefern jetzt `200 + errors[]` statt `400` (konsistent mit „not-yet-implemented"-Semantik).
- **Bewusst raus:** neue externe Adapter (CELLAR-AKN / Fedlex / eCFR / GovInfo) → **THE-439**; hierarchischer `@eId`-Output → **THE-415**.
- **Verifikation:** `source-registry.test.ts` 6, `crawl-route.test.ts` 4 (ai-act-en/dora akzeptiert, Müll abgelehnt, Array-Bounds), Provenance-Sub-Doc-Persist-Test, Crawler 113/113, `tsc` sauber. Grep-Beweis: 0 Treffer für `z.enum`-Source-Liste / `buildSourceRegistry` / duplizierte `SOURCES`.

## Teil 3 — THE-417: Ontologie-erzwungene Store-Writes (PR #44)

Der Vertragsschluss: was in den Store geschrieben wird, **muss** ontologie-konform sein — und trägt die Version, die es validiert hat.

- **Ontologie v1.2.0:** neue **`languages`-Facette** (de/en) + `framework`/`custom`-normKinds-Zeilen (Werte, die `kindFromStandardType` schon produziert → Datenzeilen, keine Code-Sonderfälle); Schema + Dupe-Check kennen die Facette.
- **Sprach-Enums kollabiert** (4. Facette, THE-413-Rezept): `RegulationLanguage` geschlossene Union → deprecated String-Alias; beide Regulation-Models validieren via null-tolerantem `isLanguage` (DELTA-4).
- **`ontologyVersion` pro Node gestempelt** (AC-2): Crawler-`$set`, beide Regulation-Models und `upsertNormDoc` — **jeder gespeicherte Fakt notiert die Ontologie-Version, die ihn validiert hat.**
- **Norm-Write-Pfad erzwungen** (schließt die offene **THE-390-P0-Pflicht**): `kind`/`jurisdiction`-Ontologie-Validators auf dem Norm-Model + `runValidators: true` auf `upsertNormDoc` (fehlte bislang).
- **Migration gehärtet** (Review-Fund): `migrate-to-norms.ts` jetzt **per-View resilient** (protokolliert Fehler, macht weiter, Exit non-zero bei Fehlern in `--apply`) + eingebauter Off-Ontologie-Pre-Check (`Regulation.distinct` vs. Ontologie-IDs) — **ein einzelnes Legacy-Doc kann die anstehende Prod-Migration nicht mehr abbrechen.**
- **AC-Abgleiche (in Linear dokumentiert):** AC-1 kein separates Ingestion-Package-Objekt — Enforcement = Mongoose-Validators (`runValidators`) + Zod-`CrawlBodySchema` (E6: keine Doppelvalidierung). AC-3 PartyRole durch Abwesenheit erfüllt (null Consumer außerhalb der Ontologie). Provenance de facto Pflicht via THE-414-Write-Site.
- **Verifikation:** Ontologie-1.2.0-Wohlgeformtheit + Facetten-Coverage 14 Tests; Sprach-Validators datengetrieben in beiden Packages (Grep `enum: ['de'` → 0); Norm-Write-Pfad end-to-end (Stamp + Off-Ontologie-Reject + framework-kind + Migrations-Resilienz), **Server-Sweep 185/185, Crawler 118/118**, `tsc` sauber (shared/server/crawler), turbo 4/4.

## Roter Faden — warum das ein „umfangreicher Tag" war

Die drei PRs sind **eine** Bewegung durch den Stack, nicht drei Einzelfeatures:

| Schicht | Vorher | Nachher (10.07.) |
|---|---|---|
| **Typen/Enums** (THE-413) | 7 handgepflegte Kopien, Drift-anfällig | 1 Ontologie-Quelle, Grep-bewiesen 0 Dubletten |
| **Verdrahtung** (THE-414) | 2 hartcodierte Registries + z.enum | 1 datengetriebene Registry + Provenance pro Fakt |
| **Store-Writes** (THE-417) | Norm-Pfad ohne `runValidators`, keine Versionsspur | Ontologie-erzwungen, `ontologyVersion` pro Node, Migration prod-sicher |

**Auszahlung schon heute (11.07.):** THE-418 konnte **DORA als reine Datenzeile onboarden — null Code** (`ca36791` / `62ad04f`). Das ist der empirische Beweis, dass die „Datenzeile-statt-Code-Edit"-These aus ADR-0004 E6 trägt. Ohne die gestrige Trilogie hätte DORA wieder 7 Enum-Edits + 2 Registry-Edits gebraucht.
