## Architektur-Entscheidungen

1. **Cache-Invalidation via Input-Hash, nicht via Service-Hooks**
   - Statt CRUD-Hooks auf Element/Connection/Mapping (komplex, error-prone) → der Hash enthält alle Inputs
   - Element-Change → element-id-set changes → hash changes → cache miss → recompute
   - Self-invalidating, kein zentraler Invalidation-Code nötig

2. **Project.settings statt Workspace.preferences**
   - BSH hat 1 Workspace, viele Projekte — Settings sollen pro Projekt sein
   - Damit kann ein Compliance-Projekt andere Gewichte haben als ein Tech-Refactor-Projekt

3. **Settings als PATCH mit Partial-Update**
   - Nur `topN` ODER nur `weights` updaten möglich (verschiedene UI-Flows)
   - Server-side weights clamping [0, 2.0] + all-zero rejection als Hard-Validation

4. **`saveCachedScores` ist fire-and-forget**
   - Wenn cache-save fail → Response geht trotzdem raus (mit fromCache:false)
   - Nur log.warn, kein 500-Fehler
