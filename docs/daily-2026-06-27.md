# 2026-06-27 — Crawler-Rettung + Korpus-Architektur entschieden

## TL;DR

Der [[Compliance-Crawler]] lief **36 Tage unbemerkt tot** („Running (unhealthy)") — heute diagnostiziert, gefixt, deployed und live verifiziert. Aus der Frage „wohin schreiben wir die Gesetze eigentlich?" wurde eine **Architektur-Entscheidung**: kanonischer Korpus, Referenz statt Kopie, dedizierte Mongo-Instanz. Zwei ADRs geschrieben, Foundation vollständig geplant + gescort. **Kein voreiliger Code.** Bau startet morgen mit Baustein A.

## Teil 1 — Crawler-Brand gelöscht (3 Layer)

Der Crawler war seit ~36 Tagen degraded. Drei getrennte Ursachen, alle gefixt:

| Layer | Bug | Fix |
|---|---|---|
| Healthcheck-Badge | BusyBox-`wget` löst `localhost`→IPv6 `::1`, Server bindet nur IPv4 → immer unhealthy | node-http-Probe auf `127.0.0.1` |
| Mongo-Reconnect | sticky `isConnected`-Flag, kein Reconnect nach Abriss | State aus `readyState` + Background-Reconnect-Loop |
| **Wahre Ursache** | Server A's `thearchitect-mongodb` published 27017 **nicht** auf den Host → Cross-Host-Crawler kam nicht ran | Sidecar `mongo-tailnet-bridge` (socat, Tailnet-only) |

- [x] Code-Fix gebaut + getestet (67/67), PR [#6](https://github.com/MaCeeeee/TheArchitect/pull/6) → master, Coolify-Redeploy
- [x] **B3:** Coolify-Port auf `100.106.223.83:3100` (Tailnet-only), public dicht
- [x] `mongo-tailnet-bridge` live → Crawler `connected:true`, Badge grün
- [x] Live-Test: 3 Gesetze / 16 §§ (LkSG, DSGVO, NIS2) gecrawlt, Qualität geprüft, Demo-Daten wieder gelöscht

**Lektion festgehalten:** Cross-Host-DB-Ports immer an die Tailscale-IP binden, nie `0.0.0.0`.

## Teil 2 — Architektur-Diskussion → Korpus-Modell

Auslöser: „Dateninkonsistenz, Datenvolumen, Serverauslastung". Kernbefund: Gesetzestext ist **Stammdaten**, wird aber als **per-Projekt-Kopie** gespeichert → Wurzel aller drei Sorgen.

Entschieden (siehe [[ADR-0001]] + [[ADR-0002]]):
1. **Referenz statt Kopie** — Mapping hält `{regulationKey, versionHash}`
2. **Geteilter Embedding-Space** — eine `regulations-corpus`-Qdrant-Collection (embed-once)
3. **Dedizierte Mongo-Instanz**, global replizierbar — Read-Replicas pro Region später ohne Re-Design
4. Klärung: Mongo = System of Record (verlustfrei), Qdrant = ableitbarer Index. Öffentlicher Korpus ist **nicht** residenz-gebunden (das ist der Tenant-Layer).
