## Geliefert

### REQ-PLATEAU-004 — Plateau-Progress-Bar full version (Track C / WSJF 71.4)

Geplant: 0.5d. Tatsächlich: ~2.5h inkl. Real-Time-Update-Architektur + manuellem End-to-End-Test.

**Backend:**
- `PlateauSnapshot` Schema erweitert um `implementedCount` / `totalChangedCount` / `implementationProgress` (Ratio 0-1) — Server-Type, läuft via `@thearchitect/shared` durch
- `plateauComputation.ts` zählt während Snapshot-Build die `implementedAt`-Flags der Wave-Elements
- Color-Band-Helper (`progressBand`, `progressColor`, `PROGRESS_BAND_COLORS`) als shared utility — selbe Thresholds wie WaveCard-Mini-Readout

**Frontend:**
- `PlateauBar.tsx` rendert pro Wave-Tab eine 48×3 px **animated Progress-Bar** unter dem Wave-Label + Counts. CSS-Transition `width 0.4s ease-out + background-color 0.4s ease-out`. Hover-Tooltip "X/Y implemented (Z%)". As-Is-Plateau hat keine Bar.
- `roadmapStore.applyImplementationToggle`-Action: patcht `activeRoadmap.waves[].elements[].implementedAt`, bumpt `roadmap.version` → memoized-Snapshot-Cache invalidiert → bei aktivem Plateau-View werden Snapshots in-place neu berechnet → PlateauBar re-rendered automatisch
- `WaveCard.tsx` ruft die Store-Action nach erfolgreichem `markImplementation` API-Call → Real-Time-Bar-Update ohne Reload

**Tweak nach UI-Review:**
- Track-Rail bei 0%-State war auf dunklem Background nahezu unsichtbar (`bg-[#1a2a1a]`). Auf `bg-[#2a3a2a] + border` aufgewertet so dass der Empty-State "tracked but not started" kommuniziert statt unsichtbar zu sein.

**Tests:** +8 in `plateauComputation.test.ts` (0%/50%/100% Bands, As-Is zero-progress, `implementedAt=null`-Filter, per-wave-not-cumulative, threshold-edge-cases). +4 in `roadmapStore.test.ts` (toggle bumpt version, un-implement, fremde-Waves-unverändert, null-Roadmap-no-op).

**End-to-End-Test auf Production validiert:** Plateau-View aktiviert → Checkbox in WaveCard getickt → PlateauBar zeigt animierte Bar sofort → Color-Übergang rot→amber→grün beim weiteren Ticken funktioniert.
